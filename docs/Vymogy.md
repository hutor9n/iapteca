# Функціональні та нефункціональні вимоги iApteca

## 1. Функціональні вимоги (FR)

**FR-01: Реєстрація користувача**
- **Опис:** Система дозволяє гостю створити обліковий запис, ввівви унікальний номер телефону, ім'я та пароль.
- **Актор:** Гість

**FR-02: Автентифікація**
- **Опис:** Користувач може увійти в систему за номером телефону та паролем. Система створює захищену сесію для доступу до ресурсів.
- **Актор:** Користувач

**FR-03: Перегляд каталогу препаратів**
- **Опис:** Система відображає список доступних препаратів з пагінацією. Дані завантажуються за допомогою SSR.
- **Актор:** Гість, Покупець

**FR-04: Пошук препаратів за назвою**
- **Опис:** Користувач може шукати ліки за текстовим запитом. Результати оновлюються динамічно.
- **Актор:** Гість, Покупець

**FR-05: Фільтрація за категоріями**
- **Опис:** Система дозволяє фільтрувати препарати за однією або декількома категоріями.
- **Актор:** Гість, Покупець

**FR-06: Перегляд детальної інформації про товар**
- **Опис:** При виборі препарату відкривається сторінка з описом, ціною, виробником та статусом наявності.
- **Актор:** Гість, Покупець

**FR-07: Додавання товару до кошика**
- **Опис:** Користувач може додати препарат до кошика. Стан кошика зберігається між сесіями.
- **Актор:** Покупець

**FR-08: Управління вмістом кошика**
- **Опис:** Користувач може змінювати кількість одиниць товару або видаляти позиції з кошика.
- **Актор:** Покупець

**FR-09: Оформлення замовлення**
- **Опис:** Система створює замовлення на основі товарів у кошику та присвоює йому статус PENDING.
- **Актор:** Покупець

**FR-10: Перегляд історії замовлень**
- **Опис:** Авторизований користувач має доступ до списку своїх минулих замовлень та їх статусів.
- **Актор:** Покупець

**FR-11: Адмін: Додавання нових препаратів**
- **Опис:** Адміністратор може додавати нові позиції до каталогу через форму в адмін-панелі.
- **Актор:** Адміністратор

**FR-12: Адмін: Редагування препаратів**
- **Опис:** Адміністратор може змінювати ціну, опис та кількість (stock) існуючих препаратів.
- **Актор:** Адміністратор

**FR-13: Адмін: Управління категоріями**
- **Опис:** Система дозволяє адміністратору створювати, редагувати та видаляти категорії ліків.
- **Актор:** Адміністратор

**FR-14: Адмін: Зміна статусу замовлення**
- **Опис:** Адміністратор може переводити замовлення в статуси CONFIRMED, COMPLETED або CANCELLED.
- **Актор:** Адміністратор

**FR-15: Адмін: Перегляд списку користувачів**
- **Опис:** Адміністратор має доступ до таблиці всіх зареєстрованих користувачів з можливістю пошуку.
- **Актор:** Адміністратор

**FR-16: Вихід із системи (Logout)**
- **Опис:** Авторизований користувач може завершити сеанс роботи, що призводить до анулювання доступу на поточному пристрої.
- **Актор:** Користувач

## 2. Нефункціональні вимоги (NFR)

**NFR-01: Security - Хешування паролів**
- **Пов'язана FR:** FR-01
- **Опис:** Паролі користувачів мають зберігатися в MongoDB виключно у вигляді хешів (bcrypt), сіль (salt) мінімум 10 раундів.
- **Метод перевірки:** Code review, інспекція БД.

**NFR-02: Security - Secure HTTP-only Cookie сесії**
- **Пов'язана FR:** FR-02
- **Опис:** Автентифікація має базуватися на захищених HTTP-only cookies. Сесія повинна бути захищена від доступу через JS (XSS) та мати термін дії 24 години.
- **Метод перевірки:** Інспекція заголовків Set-Cookie у відповіді сервера, перевірка прапорців HttpOnly та Secure.

**NFR-03: Performance - Швидкість SSR**
- **Пов'язана FR:** FR-03
- **Опис:** Рендеринг головної сторінки каталогу на сервері має відбуватися за p95 ≤ 800 мс при 50 одночасних запитах.
- **Метод перевірки:** Навантажувальне тестування (Lighthouse, k6).

**NFR-04: Performance - Затримка пошуку**
- **Пов'язана FR:** FR-04
- **Опис:** Час відповіді API на пошуковий запит має бути p95 ≤ 200 мс.
- **Метод перевірки:** Chrome DevTools Network tab, APM моніторинг.

**NFR-05: Usability - Адаптивність фільтрів**
- **Пов'язана FR:** FR-05
- **Опис:** Панель фільтрів має коректно відображатися та бути функціональною на екранах від 320px (Mobile-first).
- **Метод перевірки:** Manual UI testing на різних пристроях.

**NFR-06: SEO - Індексація карток товарів**
- **Пов'язана FR:** FR-06
- **Опис:** Кожна сторінка продукту повинна мати унікальний мета-тег Title та Description, сформовані з назви та опису препарату.
- **Метод перевірки:** SEO Audit tools, перегляд source code сторінки.

**NFR-07: Reliability - Збереження кошика**
- **Пов'язана FR:** FR-07
- **Опис:** Дані кошика в localStorage мають бути валідними протягом 30 днів після останнього додавання товару.
- **Метод перевірки:** Перезавантаження браузера, тестування після очищення сесії.

**NFR-08: Performance - Оновлення кошика**
- **Пов'язана FR:** FR-08
- **Опис:** UI має реагувати на зміну кількості товару в кошику (Zustand state update) за ≤ 50 мс.
- **Метод перевірки:** React Profiler.

**NFR-09: Reliability - Транзакційність замовлень**
- **Пов'язана FR:** FR-09
- **Опис:** При оформленні замовлення операція запису в БД та очищення кошика має бути атомарною.
- **Метод перевірки:** Unit tests на сервіс замовлень.

**NFR-10: Privacy - Ізоляція даних**
- **Пов'язана FR:** FR-10
- **Опис:** Користувач не повинен мати можливості переглянути замовлення іншого користувача, змінивши ID в URL.
- **Метод перевірки:** Negative testing API з різними токенами.

**NFR-11: Security - RBAC в Адмінці**
- **Пов'язана FR:** FR-11
- **Опис:** Доступ до методів POST/PUT/DELETE в API /admin має бути дозволений лише користувачам з `role: 'ADMIN'`.
- **Метод перевірки:** Спроба запиту з сесією звичайного клієнта (403 Forbidden).

**NFR-12: Performance - Завантаження фото**
- **Пов'язана FR:** FR-12
- **Опис:** Завантаження зображення препарату адміністратором має оброблятися (resize/compress) за ≤ 2 сек.
- **Метод перевірки:** Вимірювання часу відповіді API завантаження.

**NFR-13: Data Integrity - Цілісність категорій**
- **Пов'язана FR:** FR-13
- **Опис:** При видаленні категорії система повинна попередити адміністратора про наявність прив'язаних препаратів.
- **Метод перевірки:** Manual testing модального вікна видалення.

**NFR-14: Reliability - Автоматичне повернення стоку**
- **Пов'язана FR:** FR-14
- **Опис:** При встановленні статусу замовлення CANCELLED, кількість товарів (stock) має автоматично повернутися в БД.
- **Метод перевірки:** Integration test: Order cancellation -> Medication stock check.

**NFR-15: Privacy - Маскування номерів телефонів**
- **Пов'язана FR:** FR-15
- **Опис:** У списку користувачів для адміністратора номери телефонів мають відображатися частково маскованими (напр. +38063****123).
- **Метод перевірки:** UI testing адмін-панелі.

**NFR-16: Reliability - Очищення сесії**
- **Пов'язана FR:** FR-16
- **Опис:** Після виконання Logout сесійна cookie має бути видалена, а будь-які подальші запити до захищених ресурсів мають відхилятися сервером.
- **Метод перевірки:** Виклик Logout -> спроба доступу до /api/profile.

## 4. Зв'язок із SLI / SLO / SLA

**SLA (Service Level Agreement):**
- Система iApteca зобов'язується забезпечити доступність (uptime) на рівні **99.9%** на місяць (за винятком планових технічних робіт).
- Час реакції на критичні інциденти (P1) — до 2 годин.

**SLO (Service Level Objectives):**
- **Availability:** 99.9% успішних запитів (HTTP 2xx/3xx).
- **Latency:** 95% запитів до каталогу (FR-03) мають виконуватися за ≤ 1200 мс.
- **Search Efficiency:** 99% пошукових запитів (FR-04) мають повертати результат за ≤ 300 мс.
- **Reliability:** 100% підтверджених замовлень мають бути збережені в БД без втрати даних.

**SLI (Service Level Indicators):**
- `uptime_percentage` = (час роботи / загальний час) * 100.
- `catalog_request_duration_p95` (вимірюється через Prometheus).
- `search_latency_ms` (вимірюється на стороні API).
- `error_rate_percentage` = (кількість 5xx помилок / загальна кількість запитів) * 100.

## 5. Опис тестових сценаріїв

Нижче наведено опис 5 ключових тестів для верифікації функціональних та нефункціональних вимог системи.

### Тест 1: Валідація пошуку та продуктивності (FR-04, NFR-04)
*   **Мета:** Перевірити коректність роботи пошукового алгоритму та швидкість отримання відповіді від API.
*   **Сценарій:**
    *   **Given:** У базі даних існують препарати з назвами "Анальгін" та "Анаферон".
    *   **When:** Користувач вводить у поле пошуку запит "Ана".
    *   **Then:** Система повертає список, що містить обидва препарати. Час від моменту надсилання запиту до отримання результату не перевищує 200 мс.

### Тест 2: Безпека зберігання паролів (FR-01, NFR-01)
*   **Мета:** Гарантувати, що персональні дані користувачів (паролі) захищені за допомогою криптографічного хешування.
*   **Сценарій:**
    *   **Given:** Новий користувач заповнює форму реєстрації з паролем "SuperSecret123".
    *   **When:** Система зберігає дані користувача в базу даних MongoDB.
    *   **Then:** У відповідному полі бази даних замість "SuperSecret123" зберігається рядок (хеш), згенерований алгоритмом bcrypt. Пряме зворотне відновлення пароля з цього рядка неможливе.

### Тест 3: Атомарність створення замовлення (FR-09, NFR-09)
*   **Мета:** Перевірити цілісність процесу оформлення замовлення та автоматичне присвоєння статусу.
*   **Сценарій:**
    *   **Given:** Авторизований покупець має в кошику 2 товари.
    *   **When:** Покупець натискає кнопку "Оформити замовлення".
    *   **Then:** В базі даних створюється новий документ Order з унікальним ідентифікатором, повним переліком товарів та автоматичним статусом "PENDING". Кошик користувача очищується.

### Тест 4: Відновлення залишків при скасуванні (FR-14, NFR-14)
*   **Мета:** Верифікувати автоматичну логіку управління складом.
*   **Сценарій:**
    *   **Given:** Існує замовлення на 5 одиниць товару "Цитрамон", а його поточний залишок на складі (stock) становить 50 одиниць.
    *   **When:** Адміністратор змінює статус цього замовлення на "CANCELLED".
    *   **Then:** Система автоматично збільшує залишок "Цитрамону" в базі даних на 5 одиниць. Нове значення стоку становить 55 одиниць.

### Тест 5: Контроль доступу на основі ролей (FR-11, NFR-11)
*   **Мета:** Перевірити надійність захисту адміністративних ресурсів.
*   **Сценарій:**
    *   **Given:** Користувач авторизований у системі з роллю "CUSTOMER".
    *   **When:** Користувач намагається надіслати запит на створення нового препарату через API `/api/admin/medications`.
    *   **Then:** Система відхиляє запит, повертаючи статус помилки 403 (Forbidden). Доступ до операції блокується на рівні серверного middleware.
