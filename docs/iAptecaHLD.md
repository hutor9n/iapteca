# Glossary

| Термін | Визначення |
| :--- | :--- |
| Admin | Авторизований співробітник аптеки з роллю ADMIN, який має повний доступ до адмін-панелі: управління каталогом, замовленнями, категоріями та користувачами |
| API Route | Серверний маршрут у Next.js App Router (`/app/api/...`), що обробляє HTTP-запити на стороні сервера - не клієнтські сторінки |
| App Router | Система маршрутизації Next.js 16, яка базується на файловій структурі (`/app` directory) та підтримує React Server Components, layouts і nested routes |
| Cart (Кошик) | Тимчасовий список препаратів, доданих користувачем перед оформленням замовлення. Зберігається у стані клієнта (Zustand) - не є фінальним замовленням |
| Category (Категорія) | Тематична група препаратів у каталозі (напр. "Знеболювальні", "Вітаміни", "Антибіотики"). Адмін може створювати та редагувати категорії |
| CRUD | Create, Read, Update, Delete - чотири базові операції над даними в застосунку. Мапується на HTTP-методи: POST -> Create, GET -> Read, PUT/PATCH -> Update, DELETE -> Delete |
| Customer (Покупець) | Зареєстрований користувач з роллю CUSTOMER, який може переглядати каталог, додавати товари до кошика та оформлювати замовлення |
| Guest (Гість) | Незареєстрований відвідувач сайту, який має доступ лише до публічних сторінок (каталог, картка препарату) - не може оформляти замовлення |
| HLD | High-Level Design - архітектурний документ, що описує систему на рівні компонентів, потоків даних та бізнес-контексту без деталей імплементації |
| iApteca | Назва системи (Інтернет-Аптека) - вебплатформа для онлайн-пошуку та замовлення лікарських засобів |
| Medication (Препарат) | Лікарський засіб або медичний продукт у каталозі iApteca. Має атрибути: назва, виробник, категорія, ціна, наявність, опис, фото |
| MongoDB | NoSQL база даних, що використовується як основне сховище даних iApteca (каталог, замовлення, користувачі) |
| Mongoose | ODM (Object Data Modeling) бібліотека для MongoDB, що забезпечує взаємодію між Next.js-бекендом та базою даних |
| NFR | Non-Functional Requirement - нефункціональна вимога (швидкодія, безпека, доступність). Напр.: "Сторінка каталогу завантажується < 2 сек" |
| ODM | Object-Document Mapping - техніка, яка дозволяє працювати з документо-орієнтованою базою даних через об'єкти мови програмування |
| Order (Замовлення) | Підтверджений запит Customer на придбання препаратів. Має такий цикл: PENDING -> CONFIRMED -> COMPLETED / CANCELLED |
| OTC | Over-The-Counter - безрецептурний препарат, доступний для вільного продажу без рецепта лікаря |
| Product Card (Картка товару) | Окрема сторінка препарату з повним описом: назва, фото, ціна, опис, виробник, наявність на складі |
| REST API | Архітектурний стиль побудови API iApteca через HTTP-методи (GET, POST, PUT, DELETE). Реалізується через Next.js API Routes |
| Shadcn/UI | Бібліотека UI-компонентів, на якій побудовано інтерфейс системи |
| SSR | Server-Side Rendering - рендеринг HTML на сервері перед відправкою клієнту. Використовується для SEO-оптимізації каталогу |
| Stock (Наявність) | Кількість одиниць препарату в наявності. Якщо stock = 0 - товар відображається як "немає в наявності" |
| UI/UX | User Interface / User Experience - дизайн інтерфейсу та досвід взаємодії користувача з системою |
| Vitest | Інструмент для модульного тестування компонентів та API-маршрутів |
| Zustand | Бібліотека для управління станом на стороні клієнта (кошик, сесія користувача) |

# Business View

## Problem Statement

Більшість покупців ліків досі залежать від фізичних аптек або телефонних дзвінків, щоб дізнатися про наявність та ціну потрібного препарату. Це створює ряд конкретних проблем:

1. **Відсутність інформації в реальному часі** - покупець не може дистанційно перевірити, чи є потрібний препарат в наявності та скільки він коштує. Це призводить до марних поїздок та витрат часу.
2. **Ручна обробка запитів** - замовлення та резервування ліків відбувається телефоном або особисто. Це уповільнює процес, збільшує ймовірність помилок та прив'язує покупця до графіку роботи аптеки.
3. **Відсутність інструменту управління каталогом** - адміністратор аптеки не має зручного цифрового інструменту для швидкого оновлення цін, залишків та додавання нових препаратів.
4. **Немає фільтрації та пошуку** - без каталогу покупець не може самостійно знайти аналоги препарату, порівняти ціни або відфільтрувати товари за категорією.

Разом ці проблеми знижують доступність медикаментів, збільшують операційні витрати аптеки та погіршують клієнтський досвід.

## Solution Statement

Пропонується **iApteca** - публічна вебплатформа для пошуку та замовлення лікарських засобів з вбудованою адміністративною панеллю управління каталогом.

Платформа усуває кожну з виявлених проблем:

1. **Відсутність інформації в реальному часі** -> каталог препаратів відображає актуальні ціни та залишки з MongoDB. Покупець бачить статус наявності товару до того, як виходить з дому.
2. **Ручна обробка запитів** -> покупець самостійно формує замовлення через кошик. Система фіксує замовлення автоматично.
3. **Відсутність інструменту управління каталогом** -> адмін-панель надає адміністратору повний CRUD над препаратами, категоріями та замовленнями через веб-інтерфейс на базі shadcn.
4. **Відсутність пошуку та фільтрації** -> каталог підтримує пошук за назвою та фільтрацію за категорією, що дозволяє покупцю самостійно знаходити потрібний препарат.

Адміністратор отримує єдину точку керування асортиментом. Покупець отримує цілодобовий доступ до актуального каталогу та прозорий процес замовлення.

## System Actors

**Guest (Гість)**
- **Хто це:** будь-який відвідувач сайту без реєстрації
- **Дії:** перегляд каталогу, пошук та фільтрація препаратів, перегляд картки товару
- **Технічний рівень:** нетехнічний
- **Частота:** разово / ситуативно

**Customer (Покупець)**
- **Хто це:** зареєстрований користувач, який придбаває ліки
- **Дії:** реєстрація/логін, додавання до кошика, оформлення замовлення, перегляд історії замовлень у профілі
- **Технічний рівень:** нетехнічний
- **Частота:** регулярно

**Admin (Адміністратор)**
- **Хто це:** співробітник аптеки з правами управління системою
- **Дії:** CRUD препаратів і категорій, оновлення залишків та цін, перегляд і зміна статусу замовлень, управління користувачами
- **Технічний рівень:** нетехнічний
- **Частота:** постійно

# A. Епіки

**Epic 1 (E1): Каталог та Пошук**
- **Бізнес-мета:** Надати клієнтам можливість швидко знаходити препарати та перевіряти наявність.
- **Бізнес-цінність:** Підвищення задоволеності клієнтів, усунення необхідності дзвонити в аптеку.
- **Критерії завершення (DoD):** Код інтегровано в `main`, працює SSR, дані завантажуються з MongoDB через Mongoose, UI відповідає shadcn, немає помилок TS, тести пройдено.
- **Пов'язані Stories:** S1, S2, S3, S4, S5.

**Epic 2 (E2): Кошик та Оформлення замовлення**
- **Бізнес-мета:** Забезпечити самостійне створення замовлень покупцями.
- **Бізнес-цінність:** Автоматизація продажів, зниження навантаження на персонал.
- **Критерії завершення (DoD):** Стан кошика зберігається (Zustand + localStorage), замовлення успішно записується в MongoDB, статус PENDING встановлюється автоматично.
- **Пов'язані Stories:** S6, S7, S8, S9, S10.

**Epic 3 (E3): Управління каталогом (Адмін-панель)**
- **Бізнес-мета:** Створити інструмент для адміністратора для актуалізації асортименту.
- **Бізнес-цінність:** Забезпечення точності даних (ціни, залишки) без прямого доступу до БД.
- **Критерії завершення (DoD):** Всі CRUD операції для препаратів працюють через API Routes, доступ захищено middleware (тільки для ролі ADMIN).
- **Пов'язані Stories:** S11, S12, S13, S14, S15.

# B. Stories

**S1 (E1):** Як Гість, я хочу бачити список препаратів, щоб ознайомитись з асортиментом.
- **AC1:** Given відкрита головна сторінка, When дані завантажені, Then відображається сітка карток препаратів.
- **AC2:** Given БД порожня, When завантаження сторінки, Then відображається напис "Немає товарів".
- **AC3:** Given помилка сервера, When завантаження сторінки, Then показується відповідне повідомлення про помилку.
- **SP:** 3 | **Priority:** High

**S2 (E1):** Як Гість, я хочу шукати ліки за назвою, щоб швидко знайти потрібне.
- **AC1:** Given введено текст у поле пошуку, When натиснуто кнопку пошуку або Enter, Then список препаратів фільтрується за збігом назви.
- **AC2:** Given немає збігів за запитом, When виконання пошуку, Then відображається напис "Нічого не знайдено".
- **AC3:** Given введено спеціальні символи, When пошук, Then запит обробляється коректно без помилок сервера.
- **SP:** 3 | **Priority:** High

**S3 (E1):** Як Гість, я хочу фільтрувати за категорією, щоб бачити аналоги або групу товарів.
- **AC1:** Given обрано категорію "Вітаміни", When застосовано фільтр, Then відображаються лише товари з цією категорією.
- **AC2:** Given знято всі фільтри, When оновлення списку, Then відображаються всі доступні препарати.
- **AC3:** Given обрано кілька категорій одночасно, When застосовано фільтри, Then показуються товари з усіх обраних категорій.
- **SP:** 3 | **Priority:** Medium

**S4 (E1):** Як Гість, я хочу бачити наявність (Stock), щоб знати, чи можна замовити препарат.
- **AC1:** Given Stock > 0, When перегляд картки товару, Then відображається статус "В наявності".
- **AC2:** Given Stock = 0, When перегляд картки товару, Then відображається статус "Немає в наявності".
- **AC3:** Given Stock < 5, When перегляд картки, Then відображається попередження "Закінчується".
- **SP:** 2 | **Priority:** High

**S5 (E1):** Як Гість, я хочу бачити детальну сторінку препарату, щоб прочитати повний опис.
- **AC1:** Given клік на картку товару, When перехід, Then відкривається сторінка `/product/[id]` з детальними даними.
- **AC2:** Given введено невірний або неіснуючий ID у URL, When перехід, Then відображається сторінка 404.
- **AC3:** Given сторінка відкрита, When рендер, Then дані віддаються через SSR для SEO.
- **SP:** 3 | **Priority:** Medium

**S6 (E2):** Як Покупець, я хочу додати товар у кошик, щоб сформувати замовлення.
- **AC1:** Given Stock > 0, When клік на кнопку "Додати", Then лічильник кошика збільшується на 1.
- **AC2:** Given Stock = 0, When перегляд товару, Then кнопка "Додати" неактивна (disabled).
- **AC3:** Given товар уже є в кошику, When повторний клік "Додати", Then збільшується кількість цього товару в кошику.
- **SP:** 3 | **Priority:** High

**S7 (E2):** Як Покупець, я хочу видаляти товари з кошика, щоб коригувати свій вибір.
- **AC1:** Given товар знаходиться у кошику, When клік на іконку видалення, Then товар зникає зі списку кошика.
- **AC2:** Given останній товар видалено, When перегляд кошика, Then відображається повідомлення "Кошик порожній".
- **AC3:** Given зменшення кількості товару до 0, When збереження змін, Then товар автоматично видаляється з кошика.
- **SP:** 2 | **Priority:** High

**S8 (E2):** Як Покупець, я хочу підтвердити замовлення, щоб купити обрані ліки.
- **AC1:** Given заповнений кошик та авторизація, When клік "Оформити замовлення", Then створюється запис Order зі статусом PENDING.
- **AC2:** Given користувач не авторизований, When клік "Оформити", Then відбувається редирект на сторінку логіну.
- **AC3:** Given успішне створення замовлення, When відповідь від сервера, Then кошик автоматично очищується.
- **SP:** 5 | **Priority:** High

**S9 (E2):** Як Покупець, я хочу бачити історію своїх замовлень, щоб контролювати їх статус.
- **AC1:** Given перехід у профіль користувача, When завантаження даних, Then відображається список усіх минулих Order.
- **AC2:** Given у користувача немає замовлень, When перегляд історії, Then відображається напис "Історія замовлень порожня".
- **AC3:** Given клік на конкретне замовлення, When відкриття деталей, Then відображається повний перелік товарів та фінальна сума.
- **SP:** 3 | **Priority:** Medium

**S10 (E2):** Як Покупець, я хочу змінювати кількість товару безпосередньо в кошику.
- **AC1:** Given клік на "+", When натискання, Then кількість товару збільшується на 1, загальна сума перераховується.
- **AC2:** Given клік на "-", When поточна кількість = 1, Then кнопка "-" стає неактивною або пропонує видалення.
- **AC3:** Given клік на "+", When кількість досягла Stock, Then кнопка "+" стає неактивною.
- **SP:** 2 | **Priority:** Medium

**S11 (E3):** Як Адмін, я хочу додати новий препарат, щоб розширити каталог аптеки.
- **AC1:** Given заповнена валідна форма, When сабміт, Then створюється новий документ Medication у MongoDB.
- **AC2:** Given відсутність обов'язкових полів (назва, ціна), When сабміт, Then відображається помилка валідації.
- **AC3:** Given спроба запиту від звичайного користувача, When звернення до API, Then сервер повертає 403 Forbidden.
- **SP:** 5 | **Priority:** High

**S12 (E3):** Як Адмін, я хочу редагувати ціну та залишки, щоб підтримувати актуальність даних.
- **AC1:** Given змінена ціна препарату, When збереження форми, Then нова ціна миттєво відображається на сайті.
- **AC2:** Given зміна Stock на 0, When збереження, Then товар автоматично отримує статус "Немає в наявності" для клієнтів.
- **AC3:** Given введено невірний формат числа (від'ємне), When спроба збереження, Then поле підсвічується червоним.
- **SP:** 3 | **Priority:** High

**S13 (E3):** Як Адмін, я хочу створювати Категорії, щоб структурувати каталог.
- **AC1:** Given введено назву нової категорії, When збереження, Then вона з'являється у списку фільтрів каталогу.
- **AC2:** Given назва категорії вже існує, When спроба створення, Then відображається помилка "Така категорія вже існує".
- **AC3:** Given порожня назва, When клік "Створити", Then дія блокується фронтенд-валідацією.
- **SP:** 2 | **Priority:** Medium

**S14 (E3):** Як Адмін, я хочу змінювати статус замовлення, щоб обробляти запити клієнтів.
- **AC1:** Given замовлення у статусі PENDING, When зміна на CONFIRMED, Then статус оновлюється в БД.
- **AC2:** Given зміна статусу на CANCELLED, When збереження, Then зарезервована кількість товару повертається до Stock.
- **AC3:** Given відсутність прав адміністратора, When спроба зміни статусу через API, Then дія блокується сервером.
- **SP:** 5 | **Priority:** High

**S15 (E3):** Як Адмін, я хочу видаляти ліки з каталогу, щоб прибрати неактуальні товари.
- **AC1:** Given клік на "Видалити", When підтвердження у модальному вікні, Then запис позначається як видалений (isDeleted: true).
- **AC2:** Given скасування видалення у модалці, When закриття вікна, Then товар залишається у списку без змін.
- **AC3:** Given товар фігурує в активних замовленнях, When спроба видалення, Then відображається попередження.
- **SP:** 3 | **Priority:** Low

# C. 25 незалежних Stories
