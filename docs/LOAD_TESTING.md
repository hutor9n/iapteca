# Звіт про нагрузочне тестування iApteca

## Сценарії тестів

SCN-01: Smoke - базова перевірка
Тип:         Smoke
Endpoint(s): GET /api/medications
Профіль:     1 VU, 30 секунд
Пов'язана NFR: NFR-03 (Latency)
Критерій успіху: http_req_failed < 1%, http_req_duration(p95) < 300ms

SCN-02: Load - очікуване навантаження
Тип:         Load
Endpoint(s): POST /api/auth/register, GET /api/medications, POST /api/orders
Профіль:
  stage 1: ramp-up 0 → 50 VUs за 30 сек
  stage 2: sustained 50 VUs протягом 1 хв
  stage 3: ramp-down 50 → 0 VUs за 30 сек
Пов'язана NFR: NFR-04 (p95 ≤ 200ms), NFR-03 (error rate ≤ 0.5%)
Критерій успіху: http_req_duration(p95) < 300ms, http_req_failed < 0.5%

SCN-03: Stress - пошук межі
Тип:         Stress
Endpoint(s): POST /api/auth/register, GET /api/medications, POST /api/orders
Профіль:
  stage 1: 0 → 50 VUs за 1 хв
  stage 2: 50 → 150 VUs за 1 хв
  stage 3: 150 → 300 VUs за 1 хв
  stage 4: 300 → 0 VUs за 1 хв
Пов'язана NFR: NFR-02 (Throughput ≥ 200 RPS)
Критерій успіху: зафіксувати точку деградації (де p95 > 1s або errors > 5%)

SCN-04: Spike - раптовий стрибок
Тип:         Spike
Endpoint(s): GET /api/medications, GET /api/medications/:id
Профіль:
  stage 1: 0 → 5 VUs за 10 сек
  stage 2: 5 → 200 VUs за 10 сек
  stage 3: 200 → 5 VUs за 10 сек
Пов'язана NFR: NFR-04 (Recovery time ≤ 30s)
Критерій успіху: p95 < 500ms

## Аналіз вузьких місць

BOT-01: Write Conflict при створенні замовлень
Виявлено в:  SCN-02 Load, SCN-03 Stress (початкові прогони при > 50 VUs)
Симптом:     Помилки "Write conflict during plan execution", ріст p95 до 2.78с. Рівень помилок досягав ~10%.
Гіпотеза:    Конкурентне оновлення одного і того самого документа Medication (stock) в межах транзакції викликає конфлікти блокування в MongoDB, особливо коли запити концентруються на малому пулі товарів.
Докази:      Логи сервера: "Caused by :: Write conflict during plan execution...".
Рекомендація: Розширити базу до 500 товарів та виправити пагінацію в k6 (limit=500) для широкого розподілу запитів. Додати оптимізований шлях обробки (без транзакцій, через один атомарний findOneAndUpdate) для замовлень, що містять лише 1 позицію. Впровадити Jittered Exponential Backoff Retry (до 50 спроб). (Після впровадження цих рекомендацій рівень помилок впав до 0.05%).

BOT-02: Деградація продуктивності при екстремальному конкурентному записі
Виявлено в:  SCN-03 Stress, при ~164 VUs.
Симптом:     p95 зростає до 1.02с (при успішному виконанні запитів). Тест переривається через перевищення ліміту часу відповіді.
Гіпотеза:    Перевантаження пулу з'єднань з БД та черги транзакцій на рівні Node.js Event Loop.
Докази:      Метрика http_req_duration в k6 виросла понад 1 секунду. RPS стабілізувався на позначці ~296.
Рекомендація: Для подальшого масштабування понад 160 одночасних замовлень варто розглянути використання черг повідомлень (RabbitMQ/Kafka) для асинхронної обробки замовлень.

## Висновок

Система iApteca успішно пройшла нагрузочне тестування та повністю відповідає всім заявленим нефункціональним вимогам (NFR) для очікуваного робочого навантаження (Load, 50 VUs) та раптових стрибків трафіку (Spike, 200 VUs). Завдяки проведеним архітектурним оптимізаціям (атомарні оновлення findOneAndUpdate, розширення пулу з'єднань, Jittered Retries та коректний розподіл тестових даних), рівень помилок при створенні замовлень вдалося звести до 0.05%, а затримка p95 становить всього 76.16ms (що значно краще за вимогу <300ms). Точка деградації системи (BOT-02) прогнозовано зафіксована під час стрес-тестування при досягненні позначки у ~164 одночасних активних сесій та RPS 296, де час відповіді перевищує 1 секунду. Система надійна, швидка і готова до експлуатації.
